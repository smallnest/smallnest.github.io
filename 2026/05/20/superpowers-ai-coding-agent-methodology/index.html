Superpowers：一套让AI编程Agent拥有超能力的开发方法论

Jesse Vincent（GitHub 上的 obra）是一个在开源社区深耕二十多年的开发者。他写过键盘固件（Keyboardio 的 Kaleidoscope）、做过邮件客户端（K9-Mail 的核心维护者）、如今在 Prime Radiant 专注于 Agent 开发工具。他在 2025 年 10 月发布了 Superpowers——一个面向 AI Coding Agent 的完整软件开发方法论，以 skills 集合的形式提供，短短时间就冲到 196K stars。

如果你看过我之前写的 mattpocock/skills 那篇文章，你大概已经理解了"skill 文件"这个概念——YAML frontmatter + Markdown 指令集，Agent 读取后按指令执行。mattpocock/skills 提供的是离散的工具（盘问、TDD、架构审查），你需要自己决定什么时候用什么。Superpowers 的思路完全不同：它是一套自动触发的完整开发流程，从你打开 Agent 说出需求的那一刻起，到代码合入主分支为止，每个阶段都有对应的 skill 接管，而且 skills 之间会自动串联。

Vincent 管它叫"让你的 Agent 拥有超能力"。读完你会发现这个比喻不算夸张——它本质上是用一套指令模板，把一个散漫的 Agent 变成了一个遵循完整工程纪律的协作者。

一、核心工作流：从想法到合入的七个阶段

![01-core-workflow

Superpowers 的工作流是自动串联的，你几乎不需要手动指定"下一步用什么 skill"。下面是七个阶段的完整链路：

阶段 1：brainstorming（头脑风暴）

这是整个方法论最反直觉的部分——Agent 在写任何代码之前，必须先输出设计文档并获得你的批准。哪怕你只是让它"加一个简单的 todo list"，它也必须走完完整的头脑风暴流程。

流程是：先探索项目上下文（文件、文档、最近提交）→ 逐个提问（一次只问一个问题，优先选择题）→ 提出 2-3 个方案并推荐最优解 → 分块呈现设计方案，每块等你确认 → 写入 docs/superpowers/specs/ 目录并 commit → 自检 spec 中的占位符、矛盾、歧义 → 让你 review 最终 spec。

这里面有一条硬规则：永远不要跳过设计环节。"这个太简单了不需要设计"是 Superpowers 明确列出的反模式——简单的需求恰恰是未被检验的假设最容易造成大量返工的地方。简单项目的设计可以只写几句话，但必须写。

如果你的需求涉及 UI，brainstorming 还会提供一个 visual companion——它在浏览器里实时渲染线框图和方案对比，让你"看到"而不是"读到"设计选择。这个功能目前还比较新且耗 token，Agent 会先征求你的同意再开启。

阶段 2：using-git-worktrees（隔离工作空间）

设计获批后，Agent 不会在你的当前分支上直接动手。它会自动创建 git worktree——一个独立的并行工作目录，在新分支上做所有开发。这确保了主分支始终干净，实验性的修改不会污染现有代码。Agent 还会在 worktree 中跑项目初始化和测试基线验证，确保环境就绪。

阶段 3：writing-plans（制定实现计划）

有了设计文档，下一步是制定实现计划。这个 skill 把设计拆分为 bite-sized tasks——每个任务预计 2-5 分钟完成。每个任务包含精确的文件路径、完整的代码描述、以及验收标准。这是一种"极其详细的计划"——不是"实现用户认证模块"，而是"在 src/auth/login.ts 中创建 loginUser 函数，接收 email 和 password 参数，返回 JWT token；测试文件在 src/auth/login.test.ts"。

阶段 4：subagent-driven-development（子 Agent 驱动开发）

这是 Superpowers 最硬核的创新。它不是一个简单的"并行执行任务列表"，而是一套完整的子 Agent 编排 + 两级审查系统。

核心机制：

  1. 从计划中提取所有任务，创建完整的 todo list

  2. 对每个任务：派一个新的 subagent（独立上下文，不继承主对话历史）执行实现

  3. 实现完成后，派另一个 subagent 做 spec 合规审查——检查代码是否和 spec 完全一致，有没有漏需求或做多了

  4. 合规审查通过后，再派第三个 subagent 做 代码质量审查

  5. 两级审查都通过后，标记任务完成，进入下一个

这里有一个很关键的设计决策：两级审查的顺序不能颠倒。先检查 spec 合规性，再检查代码质量。如果倒过来，你可能已经优化了一段"不符合 spec"的代码，浪费了时间。

实现 subagent 的模型选择也有讲究：机械性任务（独立函数、清晰 spec、1-2 个文件）用便宜快速的模型；涉及多文件协调和模式匹配的用标准模型；需要设计判断力的用最强模型。这控制了审查成本。

阶段 5：test-driven-development（测试驱动开发）

TDD skill 在实现阶段自动激活，而且是 Superpowers 中最"铁血"的一个。它的核心铁律：

没有先看到测试失败，你写的生产代码必须全部删除。

不是"保留做参考"，不是"边写测试边改"，是删除。如果你在写测试之前就写了实现代码，Agent 会收到指令强制删除，从零开始。

具体执行是标准的 RED-GREEN-REFACTOR 循环：

  • RED：写一个最简测试，描述一个行为
  • 确认 RED：必须实际跑测试确认它失败了，且是因为功能缺失而失败（不是语法错误）
  • GREEN：写刚好够通过测试的代码
  • 确认 GREEN：跑测试确认通过，且所有已有测试依然通过
  • REFACTOR：只在绿灯时重构（消除重复、改善命名、提取辅助函数）

Superpowers 对"好的测试"和"坏的测试"有明确定义。好的测试名如 test('retries failed operations 3 times')——清晰描述行为，使用真实代码而非 mock。坏的测试名如 test('test1')——模糊不清，或者大量 mock 内部协作者，测的是 mock 而非真实行为。

阶段 6：requesting-code-review（代码审查）

每完成一个实现任务后，Agent 会自动发起审查。审查对照计划文件进行，按严重程度分级报告问题。致命问题会阻塞进度，必须修复后才能继续。

在 subagent-driven-development 模式下，审查不是一次性的——每个任务都会经历独立的审查循环（spec 合规 + 代码质量），全部任务完成后再做一次全局审查。这种"每步审查 + 终局审查"的双层机制，确保问题不会累积到最后的 PR review 才被发现。

阶段 7：finishing-a-development-branch（完成开发分支）

所有任务完成后，Agent 做最终验证（跑全量测试），然后给你四个选项：合并到主分支、创建 PR、保留分支待处理、丢弃分支。做完选择后，Agent 自动清理 worktree，回到主工作目录。

二、辅助 skills

![02-auxiliary-skills

除了七个核心阶段，Superpowers 还有一批辅助 skills 服务于特定场景：

systematic-debugging：四阶段根因分析流程——复现、隔离、假设、验证。不像一些 debug skill 那样给一个大而全的 checklist，它强调"一次只验证一个假设"和"防御性修复"（修 bug 的同时加固周边代码，防止同类问题再次出现）。

verification-before-completion：一个容易被忽视但很实用的 skill——在宣布"修好了"之前，强制 Agent 验证修复确实有效，且没有引入回归。

dispatching-parallel-agents：当任务之间确实没有依赖关系时，派多个 subagent 并行执行。这是一个可选的高级模式，主要用于加速独立任务的执行。

receiving-code-review：专门针对"收到审查反馈后如何正确响应"的场景。避免两种常见错误：防御性辩解和过度修复（别人提了一个小问题，你直接把整个模块重写了）。

writing-skills：创建新 skill 的元工具。如果你想把团队的内部规范打包成 skill，这个工具会引导你按 Superpowers 的标准格式编写。

三、实战：用 Superpowers 开发一个功能

![03-practical

下面用"给一个 CLI 工具添加 install 命令"为例，完整走一遍 Superpowers 的实际流程。

启动：一句话需求

你在 Claude Code 中输入：

"给我的 CLI 工具 cli-app 加一个 install 命令，能把工具安装到系统路径。"

阶段 1：brainstorming 自动触发

Agent 不会直接写代码。它检测到这是一个新功能需求，自动激活 brainstorming skill，开始盘问：

问 1："这个 install 命令应该安装到用户目录（~/.local/bin）还是系统目录（/usr/local/bin）？"
你答：用户目录优先，如果用户指定 --system 标志则安装到系统目录。

问 2："安装后需要把依赖也一起打包吗？还是只安装主二进制文件？"
你答：只安装主二进制文件，依赖已经在构建时静态链接了。

问 3："如果目标路径已经有同名文件，应该覆盖、跳过、还是提示？"
你答：默认提示用户确认，支持 --force 跳过确认。

几轮问答后，Agent 提出两个方案：

  • 方案 A：用一个 shell 脚本做安装媒介，CLI 命令生成脚本然后执行
  • 方案 B：CLI 命令直接操作文件系统

Agent 推荐方案 B（更简单、更少依赖、更易测试），你同意。然后它分段输出设计方案——先展示命令参数设计，你确认后再展示文件操作的错误处理，你确认后展示测试策略。全部确认后，Agent 把完整的 spec 写入 docs/superpowers/specs/2026-05-18-cli-app-install-command-design.md 并 commit。

阶段 2-3：隔离与计划

Agent 自动创建 worktree，切换到新分支 feature/install-command。然后激活 writing-plans skill，把设计拆成 5 个任务：

阶段 4-5：子 Agent 执行 + TDD

Agent 开始执行计划。你不需要盯着——它自己跑。以 Task 2 为例：

  1. Agent 派一个实现 subagent，指令里包含 Task 2 的完整文本、上下文说明、以及 TDD skill 的引用
  2. 实现 subagent 先写测试 test('installs binary to user bin directory')，跑测试确认失败（RED），然后写 install() 函数，跑测试确认通过（GREEN），做简单重构（REFACTOR），commit
  3. 主 Agent 派 spec 审查 subagent，检查：是否覆盖了 spec 中提到的用户目录安装行为？有没有遗漏 --dry-run 相关的需求？审查通过 ✅
  4. 主 Agent 派代码质量审查 subagent，检查：错误处理是否完善？有没有硬编码路径？函数是否过于复杂？审查通过 ✅
  5. 标记 Task 2 完成，继续 Task 3

整个过程你不管它。Agent 会连续执行，除非遇到真正需要你决策的阻塞问题。Vincent 在文档中提到，Agent 在这种模式下能自主工作一两个小时不偏离计划。

收尾：审查与合入

所有 5 个任务完成后，Agent 跑全局 code review，确认没有跨任务的集成问题。然后激活 finishing skill：

"All tasks complete. Tests pass (23/23). Options:

  1. Merge to main
  2. Create PR
  3. Keep branch
  4. Discard"

你选 2（创建 PR）。Agent 推送到 remote，创建 PR，在 PR 描述中自动附上 spec 链接、测试覆盖率变化、以及审查发现的问题列表。然后清理 worktree，回到主分支。

关键体验

整个流程中你做了三件事：（1）回答 brainstorming 的问题；（2）确认设计方案；（3）最后选择 PR 还是直接合入。中间的编码、测试、审查、修复全部由 Agent 自动完成。这不是"AI 帮你写代码"，而是"你定义要做什么，Agent 负责怎么做"。

四、与其他 skills 体系的对比

![04-comparison

如果你用过 mattpocock/skills，你会发现两者的设计哲学完全不同：

维度 mattpocock/skills Superpowers
触发方式 手动调用（/grill-me, /tdd） 自动触发（Agent 检测场景后自行决定）
Skills 关系 独立使用，自由组合 严格串联，阶段之间有硬依赖
设计阶段 /grill-with-docs 盘问 brainstorming 强制设计文档
实现方式 主对话中逐个 task 执行 子 Agent 编排 + 两级审查
TDD 纪律 建议垂直切片 铁律：代码先于测试就删除
适用场景 灵活开发，随时取用 完整功能开发，端到端流程

没有哪个更好——如果你的工作流偏灵活探索，mattpocock/skills 的工具箱式设计更合适。如果你在做一个明确的功能开发，Superpowers 的全程自动化更能保证质量。

五、冷眼旁观：Superpowers 的承诺与局限

![05-critique

如果说前面的章节是在讲 Superpowers 能做什么，这一节我们要问一个更尖锐的问题：它没有缺点吗？

设计的代价：当流程变成负担

Superpowers 对"先设计再实现"的坚持，在软件工程教科书里能找到完美的理论支撑。但实际开发中，很多有价值的东西是通过"先做出来看看"发现的。你没办法在 brainstorming 阶段预料到用户会怎么用这个功能，因为你自己也没用过它。

Vincent 在设计文档中提到"简单的项目可以写几句话"，试图给敏捷留出空间。但实践中，Agent 并不擅长判断"什么叫简单"——它倾向于把一切需求都展开成完整的 brainstorming 流程，因为 skill 告诉它"不要跳过设计"。一个五分钟能写完的 bug 修复，可能要花三分钟回答 Agent 的盘问、两分钟确认设计文档、然后才轮到你期待的那五分钟。对于熟练的开发者而言，这种体验可能更像是"终于教会了 Agent 怎么做事"而非"Agent 帮我把事做了"。

这不是反对设计。这是提醒你：Superpowers 的设计纪律是有摩擦成本的。它最适合的场景是那种"需要多人协作、多天开发、多轮审查"的中大型功能——这种场景下，前期设计投入被后期返工避免所抵消。但对于个人开发者的快速迭代、原型探索、或者"先做个 MVP 看看反响"的项目，Superpowers 的全程流程可能是一种过度工程。

谁在为这套方法论买单？

Superpowers 的 token 消耗模型值得算一笔账。以 subagent-driven-development 模式为例，每个任务触发至少三次 Agent 调用（实现 + spec 审查 + 代码审查），如果有问题还要走审查修复循环。对于一个 10 个任务的中等功能，保守估计 30-50 次 Agent 调用。用小模型跑机械任务可以降低成本，但小模型在 spec 理解和代码生成上的质量下降会反过来增加修复循环的次数——成本可能只是从 token 账单转移到了你的等待时间上。

这引出了一个更根本的问题：Superpowers 的受众到底是谁？ 是那些已经熟练掌握传统软件工程方法、现在想让 Agent 替代自己执行重复性工作的资深开发者？还是那些缺乏工程经验、需要一套自动化流程来弥补纪律缺失的新手？Vincent 在 README 中说这套流程让 Agent "像一个有热情但品味差、缺乏判断力、不熟悉项目上下文、还讨厌写测试的初级工程师"。问题是——如果你自己都不知道怎么带一个初级工程师，你有多大把握带好一群 AI 初级工程师？

这不是怀疑 Superpowers 的价值。196K 的认可不可能全是盲目的。这是说，Superpowers 在"降低门槛"这件事上可能做了一个错误的承诺。它降低的是 Agent 写代码的组织门槛——不用手动组织 subagent、不用人工审查每个任务的 spec 合规性。但它并没有降低——甚至可能提高了——对使用者的工程判断力要求。你需要判断 Agent 的设计方案是否合理、需要决定哪些审查意见是有效的、需要在 subagent 报 BLOCKED 时诊断根因。这些能力恰恰是传统软件工程中最难培养的部分。

Superpowers 是一个值得认真对待的工程实验。但对待它的最好方式，也许不是把它当作答案，而是把它当作一个起点——理解它的设计意图，然后根据自己的项目和能力，裁剪出一套属于你自己的流程。就像 Vincent 在文档末尾写的那样，这不是宗教。这是一套工具。工具的意义不在于被信仰，而在于被使用、被修改、被超越。

六、FAQ

Q1：Superpowers 的 brainstorming 是不是太"重"了？给一个 todo list 也要走完整流程？

Superpowers 的立场很明确：简单的需求恰恰是最容易因为未检验假设而翻车的地方。"加一个 todo list"——存储在哪？本地还是云端？需要同步吗？排序规则是什么？这些假设如果在实现中途才发现不对，返工成本远超 brainstorming 那几分钟。但设计文档的长度可以按复杂度缩放——简单的需求写几句话就行，不需要长篇大论。

Q2：子 Agent 的两级审查（spec + code quality）会不会太费 token？成本怎么控制？

这是 Superpowers 最明显的成本来源——每个任务要派 1 个实现 agent + 2 个审查 agent。但它采用了"用最便宜的模型做最简单的事"的策略：机械性实现任务用小模型，审查任务用标准或强模型。而且 Vincent 的核心理念是"早发现比晚修复便宜"——在子 Agent 阶段拦截问题，比合并后 debug 省 token。实际使用中，只要任务拆得够细（2-5 分钟一个），单任务的审查成本是可控的。

Q3：Agent 在 subagent-driven-development 模式下真的能自主跑一两个小时？会不会跑偏？

会跑偏，但有机制防范。第一道防线是 spec——每个任务都对照 spec 验证合规。第二道防线是两级审查——如果实现 subagent 的理解有偏差，审查 subagent 会挡下来。第三道防线是实现 subagent 有四种状态报告（DONE / DONE_WITH_CONCERNS / NEEDS_CONTEXT / BLOCKED），遇到问题会主动升级而不是硬撑。Vincent 提到"一两个小时"是比较顺利的情况，实际使用中还是会时不时需要你介入回答 subagent 的问题。

Q4：TDD 的"事先写的代码必须删除"这条铁律是不是太极端了？

这不是道德洁癖，是一个实用的工程判断。测试后写的代码有一个根本问题：你永远不知道测试是否真正捕捉到了正确的行为——因为测试一写就通过。只有先看到测试因为"功能缺失"而失败（不是语法错误），你才能确认这个测试确实在测你要实现的东西。如果保留了先写的代码再做测试，你实际上是在"对着实现写测试"而非"对着需求写测试"，很容易测的是实现细节而非行为。

Q5：Superpowers 适合老项目还是新项目？

都适合，但老项目需要注意一点：brainstorming 的"探索项目上下文"阶段会读取现有文件、文档和最近提交来理解代码库。如果老项目的代码结构混乱、缺乏模式，Agent 可能需要更长的探索时间，提出的方案也可能不够精准。建议在老项目中先用一次 brainstorming 走完整流程，看看 Agent 对代码库的理解程度，再决定是否全面采用。

Q6：多个 Agent 平台都支持，实际体验有差异吗？

有。Superpowers 的 skills 是通用的 Markdown 指令，但不同 Agent 对 skill 的"理解深度"和"执行纪律"不同。Claude Code 对 subagent 的支持最成熟（Superpowers 最早就是为 Claude Code 设计的），Codex CLI 的执行速度更快但 TDD 纪律偶尔会松动。实际使用中建议用 Claude Code 做主 Agent、Codex 做实现 subagent 的混合模式。

Q7：我可以只装部分 skill 吗？比如只用 brainstorming 和 TDD，不用子 Agent 编排？

技术上可行——skills 是独立的 .md 文件，你可以选择性安装。但 Superpowers 的很多 skill 之间有硬依赖：subagent-driven-development 依赖 writing-plans 产出的计划格式，writing-plans 依赖 brainstorming 产出的 spec 路径约定。拆开用的结果可能是 skill 之间的衔接断裂，导致流程中断。如果你只需要部分功能，更推荐用 mattpocock/skills 那种独立组合式的 skill 集合。

Q8：和 mattpocock/skills 能共存吗？

可以，但需要留意冲突点。两者的 TDD skill 都会在你写代码时被触发，可能产生竞争。建议的做法是：用 Superpowers 作为主要工作流（覆盖 brainstorming → 实现 → 合入的完整链路），把 mattpocock/skills 中的 /diagnose、/improve-codebase-architecture、/prototype 等 Superpowers 没有覆盖的 skill 作为补充工具。两者用的都是 skill 文件格式，在同一个项目中安装不会相互覆盖。
